# 11 数据缺口与补齐优先级

更新时间：2026-05-28

本文档记录当前数据仓库距离 V1 可开发、可测算、可上线之间的缺口。

结论：当前仓库已经有数据骨架、2025 年第一批参数、退休年龄规则初版，以及三城 2015-2025 关键参数工作台。距离可用于养老金测算仍有缺口，尤其是个人账户记账利率策略、深圳历史参数、上海候选数据复核和样例测试集，仍是 V1 之前必须处理的问题。

2026-05-28 补充决策：V1 首开城市确定为上海。数据补齐和上线验收优先围绕上海闭环；北京、深圳后续再开放，不能因为上海进入首开准备就放宽其数据闸门。

## 一、三城历史年度参数缺失最大

当前 `data/official/city-history-coverage.json` 显示 canonical 历史参数库仍未完全归并：

- 上海：`city-key-params-2015-2025.json` 已有 2015-2025 工作台数据；2020、2021、2024、2025 有官方来源记录，2015-2019、2022-2023 仍为候选值。
- 北京：`city-key-params-2015-2025.json` 已有 2015-2025 缴费基数官方历史表数据，2025 养老待遇计发基数已记录。
- 深圳：`city-key-params-2015-2025.json` 已有 2024-2025 深圳/广东官方来源数据，2015-2023 仍是部分回补。

也就是说，2015-2025 的“关键参数工作台”已经建立，但还没有全部归并为 production-ready canonical 参数库。

下一批建议回补：

- 2015-2024

原因：

- 这是最近十年的核心缴费期。
- 很多接近退休用户的后半段缴费记录会落在这个区间。
- 数据相对更容易在地方人社局公告中找到。

## 三、退休年龄规则表缺失

已新增可计算的本地规则表初版：

- `data/official/retirement-age-rules.json`
- 按出生年月、性别/岗位类型推算退休年龄的规则。
- 退休年份对应最低缴费年限规则。

重要性：

- 这是 V1 快速测算的硬依赖。
- 如果没有退休年龄，就无法确定预计退休年份。
- 如果无法确定退休年份，就无法选择计发月数、最低缴费年限和未来缴费年限。

剩余动作：

- 用国家社保公共服务平台退休年龄计算器做抽样校验。
- 明确是否支持特殊工种、病退、弹性提前/延迟退休；V1 当前不建议支持个案审批类场景。

## 四、个人账户记账利率缺失

当前三城都缺：

- `accountInterestRate`

影响：

- 用户不知道个人账户余额时，无法较准确估算个人账户累计额。
- 历史个人账户累计额无法逐年滚动计算。
- 个人账户养老金误差会增大。

V1 必须确定策略：

- 优先方案：维护官方历年个人账户记账利率。
- 降级方案：采用保守默认值，并显著降低准确度等级。
- 推荐交互：鼓励用户填写个人账户余额，绕过历史账户倒推误差。

## 五、过渡性养老金数据缺失

当前还没有三城完整历史政策分段。

缺：

- 视同缴费规则。
- 统账结合前后政策节点。
- 上海、北京、深圳各自过渡性养老金口径。
- 早年工作经历如何影响结果。

V1 建议：

- 不估金额。
- 结果页展示“过渡性养老金暂未纳入”。
- 如果用户有早年工作经历，提示“实际养老金可能高于当前估算，最终以社保经办机构核定为准”。

## 六、benchmark 样本未生成

当前 `data/generated/benchmark-meta.json` 状态为：

```json
{
  "status": "not_generated",
  "benchmarkVersion": null
}
```

缺：

- `benchmark-samples.json`
- 样本生成规则。
- 三城固定基准样本。
- 竞争力分数校验样例。

影响：

- 现在不能上线“超过 X% 同城估算样本”。
- 否则竞争力分数缺乏稳定基准。

## 七、复核状态未完成

当前很多来源和参数仍是：

- `pending_second_review`
- `pending_local_review`
- `pending_city_applicability_review`

需要统一口径：

- `data/README.md` 已调整为“已维护第一批官方来源数据，仍处于待二次复核状态”。后续仍需保持所有文档一致，避免把“已采集”误写成“生产可用”。
- 数据状态必须区分“已采集”“已复核”“可生产使用”。

推荐状态：

- `collected`
- `pending_second_review`
- `reviewed`
- `production_ready`
- `deprecated`
- `needs_city_applicability_review`

## 八、样例测试数据

已新增：

- `data/generated/retirement-age-test-cases.json`
- `data/generated/calculation-test-samples.json`
- `data/generated/generate-test-samples.mjs`

当前覆盖：

- 414 条退休年龄规则穷举样例。
- 216 条 MVP 养老金测算输入矩阵样例。
- 1446 条扩展养老金测算输入矩阵样例。

用途：

- 验证公式数量级。
- 验证城市参数是否选对。
- 验证退休年龄规则。
- 验证误差提示。
- 验证竞争力分数不会反直觉。

当前限制：

- 暂未包含养老金金额 expected value。
- 暂未包含竞争力分数 expected value。
- 金额类 expected value 需要在个人账户记账利率策略、城市参数复核和计算公式实现后生成。
- 第一版开发优先使用 MVP 样例，扩展样例不代表首版必须覆盖所有特殊场景。

## 九、建议补齐优先级

建议顺序：

1. 退休年龄规则表抽样复核。
2. 上海 2015-2025 关键参数二次复核和 production-ready 标记。
3. 上海个人账户记账利率策略，或明确“用户填写账户余额优先”的降级口径。
4. 上海金额 expected value 测试样例复核。
5. 微信开发者工具和真机完整验收。
6. 北京、深圳 2015-2025 关键参数补齐和复核。
7. 深圳缴费基数适用性复核。
8. benchmark 样本。
9. 2000-2014 历史参数。
10. 过渡性养老金历史政策分段。

## 十、对优先级的补充判断

该优先级基本合理，但需要注意两个细节：

- 退休年龄规则表确实是 V1 硬依赖，应优先于历史参数，因为它决定退休年份、计发月数和最低缴费年限。
- benchmark 样本不应该在历史参数和样例测试之前生成，否则样本会随着参数修正频繁失效。

## 十一、V1 最低可开发门槛

在真正开始开发计算逻辑前，最低应具备：

- `retirement-age-rules.json` 初版，且完成抽样校验。
- 三城 2025 参数完成二次复核。
- 三城至少 2015-2025 关键参数初版，并明确哪些字段是 candidate。
- 个人账户记账利率降级策略。
- 10-20 个样例测试集。
- 明确哪些结果是快速估算，哪些是较准确估算。

## 十二、基础 V1 最低可上线门槛

上线前最低应具备：

- 首开城市上海 2015-2025 关键参数通过复核。
- 退休年龄规则表通过样例测试。
- 个人账户余额未知时有明确误差提示。
- 过渡性养老金不纳入金额计算，并有明确提示。
- 结果页免责声明和数据版本展示。
- 微信开发者工具真实编译通过。
- 真机完整验收通过。

如果某个城市未达到以上门槛，该城市不应向普通用户开放测算。

## 十三、竞争力分数上线门槛

竞争力分数不是基础 V1 的硬依赖。基础测算可以先上线，但只要 benchmark 未通过校验，结果页和分享页就不能展示竞争力分数、百分位或“超过 X% 同城样本”。

竞争力分数上线前最低应具备：

- `benchmark-samples.json` 已生成。
- `benchmark-meta.json` 有明确 `benchmarkVersion`。
- benchmark 使用的城市参数版本已通过复核。
- 竞争力分数样例测试通过。
- 已检查低缴费、短年限用户不会因为模型异常获得反直觉高分。
- 分享文案明确是“同城估算样本”，不写成真实用户排名。
