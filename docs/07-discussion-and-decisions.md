# 07 沟通过程与决策记录

更新时间：2026-05-28

## 初始想法

用户最初想做一个“中国退休工资计算”的小程序，但对计算方式不熟悉。

已知影响因素包括：

- 缴纳社保的总月份。
- 每年缴纳社保水平。
- 缴纳地区。
- 多地缴纳。
- 当地社会平均工资。
- 地区差异。

随后讨论明确：第一版应该聚焦城镇职工基本养老保险估算。

## 模块规划讨论

用户提出正常来说还会涉及：

- 机关事业
- 城乡居民养老
- 企业年金
- 职业年金
- 个人养老金

讨论后确认首页可以规划为 6 个模块，但 V1 只做城镇职工。

这样既保留长期结构，又避免 V1 过重。

产品评审后一度建议 V1 首页只展示城镇职工主入口。后续确认改为：V1 首页展示 6 个模块，其中城镇职工可点击，机关事业、城乡居民、企业年金、职业年金、个人养老金置灰展示，并标记“敬请期待”。

## 产品定位讨论

一开始有“计算器”倾向，后来讨论认为它可以升级成“低门槛养老规划工具”。

用户进一步明确：当前小程序前期定位仍应是工具化，不要马上扩展成完整养老规划。

最终定位：

```text
退休金算盘是一个低门槛养老金测算工具。
```

后续可以在工具基础上迭代养老规划和周边能力。

## 输入复杂度讨论

用户明确提出：

1. 尽可能让用户少填写。
2. 必要信息必填，其他信息选填。
3. 复杂数据可以折叠起来。
4. 公共数据由产品采集维护，不让用户自己查。
5. 功能可以大而全，但使用复杂度要低。

因此形成核心原则：

```text
少填也能算，细填更准确。
```

## V1 支持范围确认

用户确认：

- V1 目标支持上海、北京、深圳，但城市是否开放取决于该城市是否达到数据上线门槛。
- 以单城市快速估算为主。
- 多地缴费不作为 V1 主流程，V1 仅提示这会降低结果精度。
- 用户群体主要定位为 40 岁以上、接近退休人群。
- 结果页展示估算范围，不使用“统计误差”口径。
- 计算结果拆成总数和各维度数据展示。
- 增加“补充个人账户余额，让结果更准”的引导。

## 当前缴费水平讨论

用户不确定应该让用户填月薪、缴费基数，还是 60%/100%/300% 档位。

讨论后确定给多种入口：

- 不清楚，按当地最低档估算。
- 按税前月薪估算。
- 我知道社保缴费基数。
- 按社平工资档位估算。

这样可以兼顾低门槛和准确性。

## 个人账户余额讨论

用户问是否允许用户不知道个人账户余额。

确认：

- 允许不知道。
- 不知道时系统用估算值。
- 可以同时展示“估算值”和“用户填写值”的区别。
- 如果用户填写个人账户余额，结果更准确。
- V1 只把个人账户余额作为重点补充项；历史缴费、未来计划、早年工作经历等复杂补充项后置。

## 已退休校验场景讨论

用户问是否支持“我已经退休了，帮我校验养老金是否合理”。

确认：

- V1 暂时不做。
- 这是另一个产品流程，后续再考虑。

## 登录和保存数据讨论

一开始用户提到需要缓存历史测算，后来补充明确：

- 前期做成纯前端小程序。
- 不做登录。
- 不保存云端数据。
- 可以本地缓存。

最终确认：

- V1 纯前端可行。
- 本地缓存最近一次测算。
- 不上传个人数据。

## 官方参数库讨论

用户希望公共数据由产品维护，例如城市社平工资数据。

确认：

- V1 参数可以打包在小程序代码中。
- 数据一年或阶段性变动一次，前期人工维护可行。
- 后续可做半自动采集或后台维护。
- 每个参数必须记录官方来源 URL。

## 竞争力和攀比心理讨论

用户希望结果页增加吸引力，例如“您的退休金打败全国多少用户”。

讨论后认为：

- V1 没有真实用户数据，不能宣称真实打败全国用户。
- 可以用固定 benchmark 样本计算“超过同城估算样本”。
- 分享出去后，分数必须稳定、可解释。
- 不能出现缴纳更少、年限更短却明显分数更高的反直觉情况。

产品评审后修订：竞争力分数不作为基础 V1 的硬依赖。只有在 benchmark 样本生成、样例测试通过并版本化后，才展示竞争力分数和百分位。

后续评分模型仍按多维度设计：

- 金额水平
- 缴费贡献水平
- 缴费年限水平
- 个人账户积累水平

并加入保护规则。

## 结果页视觉讨论

用户希望结果页有烟花、奖杯等效果。

确认：

- 可以做分层视觉，但 V1 第一屏优先展示金额区间、拆分、关键假设、数据质量和免责声明。
- 高分可显示小奖杯 / 徽章和局部轻量动效，属于竞争力分数通过闸门后的增强能力。
- 中等分可显示徽章和轻量彩带，同样后置。
- 低分给温和鼓励。
- 风格要可信，不要过度游戏化，不做全屏烟花或长时间循环动画。

## 分享需求讨论

用户希望用户分享小程序，带动朋友参与测算养老金。

确认：

- V1 分享默认展示品牌和普通测算入口，不展示养老金金额。
- 竞争力分数和百分位只有在 benchmark 通过校验后才展示。
- 默认不展示养老金金额，保护隐私。
- 后续可提供“展示金额”开关，但不进入 V1 原型验收范围。

## 需要拍板的问题

用户最终拍板：

1. 快速测算字段确认。
2. 当前缴费水平输入方式确认。
3. V1 只把个人账户余额作为重点补充模块，其他复杂补充后置。
4. 过渡性养老金 V1 暂不精算确认。
5. 允许本地缓存确认。
6. 竞争力分数采用多维度模型，但必须等 benchmark 样本生成并通过校验后再上线。
7. 分享图选择 B：默认不展示金额；无 benchmark 时也不展示竞争力排名。
8. 官方参数库打包在小程序代码中确认。

## 首开城市和发布节奏确认

2026-05-28 追加确认：

- 首开城市选择上海。
- 北京、深圳继续在首页和城市选择中展示，但未达到数据上线门槛前不开放普通测算。
- 下一阶段进入微信开发者工具联调和真机完整验收。
- 真机验收应覆盖：大字号、键盘弹起、底部安全区、长文案换行、分享入口、账户余额补充、内部预览结果警示。
- 一期不接腾讯广告；广告位和商业化可以后续评估，但不进入 V1 发布范围。
- 灰度发布应在上海数据复核、金额样例验收、微信开发者工具和真机验收全部通过后再考虑。

## 命名确认

最终名称：

中文名：退休金算盘

英文名：Pension Abacus

名称判断：

- 有工具属性。
- 比“养老金计算器”更有记忆点。
- “算盘”带有计算、估算、传统可信感。
- 英文名与中文含义对应。
