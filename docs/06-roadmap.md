# 06 功能规划与迭代路线

更新时间：2026-05-28

## 总体规划

退休金算盘先做工具化测算，再扩展周边能力。

产品不应一开始就做成“大而全养老规划平台”，否则会导致输入复杂、结果不可信、开发周期过长。

推荐路线：

```text
V1：可信快速测算 MVP
V1.1：竞争力分数与分享优化
V2：多城市、多地缴费、参数后台
V3：其他养老金模块
V4：养老规划和周边能力
```

## V1：可信快速测算 MVP

目标：

- 用户能快速估算城镇职工养老金。
- 首开城市确定为上海；北京、深圳继续展示为规划城市，但必须达到数据上线门槛后才开放。
- 支持低门槛输入。
- 支持结果拆分、估算范围、关键假设、数据质量和免责声明。
- 支持普通分享。

功能：

- 首页
- 城镇职工快速测算
- 结果页
- 补充个人账户余额
- 未来缴费默认按当前缴费水平持续到退休
- 过渡性养老金暂未纳入提示
- 本地缓存
- 普通分享卡片
- 官方参数 JSON 打包
- 数据来源 / 关于 / 免责声明页面
- 退休年龄规则表
- 样例测试集

不做：

- 登录
- 服务器
- 真实排行榜
- 一期不接腾讯广告或其他商业化广告位
- 已退休校验
- 全量多地缴费精算
- 详细历史缴费分段录入
- 早年工作 / 视同缴费金额估算
- 未通过 benchmark 校验的竞争力分数

## V1.1：体验与准确度增强

目标：

- 降低误差。
- 提升分享转化。
- 增强用户信任。

可做：

- 固定 benchmark 样本生成和校验。
- 退休金竞争力分数。
- 竞争力分享卡片。
- 估算状态标签增强：快速估算 / 账户余额已补充 / 数据待补齐。
- 后续可在数据质量模型成熟后探索准确度条，但不展示伪精确百分比。
- 数据缺失提示：最影响准确度的 1-2 个字段。
- 结果页解释：“为什么是这个数”。
- 城市参数说明页。
- 分享海报美化。
- 个人账户余额查询指引。
- 历史缴费记录导入模板或手动分段填写。

## V2：城市与政策能力增强

目标：

- 从单城市快速估算走向更真实的多地场景。

可做：

- 增加更多城市。
- 多地缴费经历计算。
- 待遇领取地规则提示。
- 灵活就业人员测算。
- 参数远程配置。
- 参数管理后台。
- 版本化参数更新。
- 更完整的退休年龄规则表。

## V3：扩展养老金模块

首页六个模块逐步补齐。

模块：

- 机关事业养老金测算
- 城乡居民养老金测算
- 企业年金测算
- 职业年金测算
- 个人养老金测算

建议顺序：

1. 城乡居民养老
2. 个人养老金
3. 企业年金
4. 职业年金
5. 机关事业

原因：

- 城乡居民和个人养老金相对更适合工具化。
- 企业年金/职业年金需要更多账户数据。
- 机关事业规则复杂，历史分段和职业年金耦合较强。

## V4：养老规划能力

在工具化测算跑通后，再考虑养老规划。

可做：

- 退休收入结构分析。
- 缺口测算。
- 补充养老金建议。
- 个人养老金缴费模拟。
- 商业养老保险对比。
- 家庭养老预算。
- 退休城市生活成本估算。

## 商业化可能性

短期不建议强商业化，先建立工具信任。

一期发布不接广告。腾讯广告可以作为后续商业化能力预留，但必须等基础测算可信、用户完成率和复访路径稳定后再评估，且不应干扰首页主入口和结果页主结果。

长期可能：

- 养老规划报告。
- 企业 HR 工具。
- 金融/保险线索，但必须合规且谨慎。
- 城市社保政策知识库。

## 关键里程碑

建议里程碑：

1. 完成 PRD 和原型。
2. 完成退休年龄规则表。
3. 完成上海 2015-2025 关键参数回补、复核和金额样例验收，作为首开城市。
4. 完成养老金计算模型验证。
5. 微信开发者工具真实编译通过。
6. 真机完整验收通过，覆盖大字号、键盘、安全区、长文案、分享和账户余额补充。
7. 完成小程序 V1 前端冻结。
8. 上线灰度版本。
9. 生成并校验固定 benchmark 样本后，再开启竞争力分数。
