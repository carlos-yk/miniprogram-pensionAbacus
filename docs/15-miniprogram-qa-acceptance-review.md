## 四、P1 高优先级问题

### 1. 结果页缺少可见分享入口

位置：

- [result.wxml](../miniprogram/pages/result/result.wxml)
- [result.js](../miniprogram/pages/result/result.js)

问题：

结果页已有 `onShareAppMessage`，但页面上没有可见的“分享给朋友”按钮或分享卡片。用户无法明确触发分享。

修复状态：

- 已修复。
- 结果页已增加 `open-type="share"` 的“分享给朋友”按钮。
- 默认分享仍复用隐私安全 payload。

建议：

- 增加 `open-type="share"` 的分享按钮。
- 按 V1 隐私规则复用当前安全分享 payload。
- 分享卡默认不展示金额、出生年月、缴费年限、个人账户余额。

### 2. 缴费档位缺少 150%

位置：

- [calculate.js](../miniprogram/pages/calculate/calculate.js)

问题：

文档要求社平工资档位包含 60% / 100% / 150% / 300%，当前实现只有 60% / 100% / 300%。

修复状态：

- 已修复。
- 缴费档位已补齐 150%。
- 已增加自动测试覆盖 60% / 100% / 150% / 300%。

建议：

- 补充 150% 档位。
- 增加测试覆盖，防止输入项与文档再次偏离。

### 3. 计算逻辑仍按当前年份参数估算历史缴费

位置：

- [pensionCalculator.js](../miniprogram/utils/pensionCalculator.js)

问题：

当前测算逻辑使用当前年份参数估算过去和未来缴费。虽然当前三城数据闸门均阻止正式提交，但如果未来只把城市状态改成 ready，仍会违反“历史年度不能静默使用最新年份参数”的设计原则。

修复状态：

- 已修复闸门风险。
- ready 城市仍必须通过历史参数覆盖检查；历史覆盖不完整时会降为“历史参数补齐中”，不可提交。
- 已增加自动测试覆盖“ready 但历史覆盖不完整仍不可算”。

建议：

- ready 闸门必须绑定历史参数覆盖状态。
- 或先实现按年度展开 / 明确降级策略后，再开放城市测算。
- 在数据未完整前，不能仅通过改城市状态开放普通用户测算。

## 五、P2 中优先级问题

### 1. 提交按钮视觉置灰但仍绑定 submit

位置：

- [calculate.wxml](../miniprogram/pages/calculate/calculate.wxml)

问题：

按钮视觉上置灰，但仍绑定 `submit`。当前 `submit` 内部会二次校验，所以不会产生结果，但交互层不够干净。

修复状态：

- 已修复。
- 置灰按钮改为绑定 `onSubmitTap`，不可提交时只展示原因，不进入测算。

建议：

- `!canSubmit` 时绑定专门的不可用处理。
- 或给出“暂未开放测算 / 查看数据来源”的明确提示。

### 2. 城市状态枚举与 UI 文档未完全一致

位置：

- [dataGate.js](../miniprogram/utils/dataGate.js)

问题：

UI 文档包含“历史参数补齐中”，当前深圳显示为“适用性复核中”。功能上可接受，但视觉验收口径不完全一致。

修复状态：

- 已修复。
- 深圳当前展示为“历史参数补齐中”。

建议：

- 补齐 UI 文档中的状态枚举。
- 明确深圳当前使用“适用性复核中”还是“历史参数补齐中”。

### 3. 自动测试尚未覆盖页面级交互

位置：

- [tests](../tests)

问题：

当前测试覆盖核心工具函数、配置和文件完整性，但没有覆盖页面交互、分享按钮可达、账户页回流、WXML 真实编译。

建议：

- 在微信开发者工具做页面级联调。
- 后续可增加小程序自动化或截图级验收。

### 4. `female_unknown` 结果页未展示退休月份区间

位置：

- [result.js](../miniprogram/pages/result/result.js)

问题：

结果页展示了退休年龄区间，但没有同时展示退休月份区间。

修复状态：

- 已修复。
- `female_unknown` 结果页同时展示退休年龄区间和退休月份区间。

建议：

- 对“不确定，帮我估算”展示退休年龄和退休月份的区间。
