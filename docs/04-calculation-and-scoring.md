# 04 计算模型与竞争力评分

更新时间：2026-05-28

## V1 计算范围

V1 计算城镇职工基本养老保险的估算值。

主要包括：

- 基础养老金
- 个人账户养老金
- 过渡性养老金提示

V1 不做官方精算，不替代社保经办机构核定。

## 城镇职工基本养老金常见结构

通常可拆分为：

```text
基本养老金 = 基础养老金 + 个人账户养老金 + 过渡性养老金
```

其中：

- 基础养老金与养老金计发基数、本人平均缴费指数、缴费年限相关。
- 个人账户养老金与退休时个人账户累计储存额和计发月数相关。
- 过渡性养老金与早年工作、视同缴费、地方政策相关，地区差异较大。

## 基础养老金估算公式

V1 建议采用常见估算公式：

```text
基础养老金 = 养老金计发基数 × (1 + 本人平均缴费指数) ÷ 2 × 累计缴费年限 × 1%
```

字段说明：

- 养老金计发基数：由地区年度公布，上海、北京、深圳需分别维护。
- 本人平均缴费指数：根据历年缴费基数与当地社平工资或计发相关基准的比例估算。
- 累计缴费年限：用户填写的已缴费年限 + 未来继续缴费年限。

## 个人账户养老金估算公式

```text
个人账户养老金 = 退休时个人账户累计储存额 ÷ 计发月数
```

字段说明：

- 个人账户累计储存额：优先使用用户填写的个人账户余额。
- 如果用户不知道个人账户余额，则按缴费基数、个人缴费比例、缴费年限、记账利率进行估算。
- 计发月数：根据退休年龄查表，例如 50 岁、55 岁、60 岁、65 岁对应不同月数。

## 过渡性养老金处理

过渡性养老金通常与视同缴费年限、早年工作经历和地区政策相关。

V1 策略：

- 默认不精算。
- 结果页展示“过渡性养老金：暂未纳入”。
- 如果用户存在早年工作 / 视同缴费经历，结果页提示实际养老金可能高于当前估算。
- 即使用户填写早年工作 / 视同缴费信息，V1 也不展示该项金额，只提示实际养老金可能受影响。
- 明确说明最终以社保经办机构核定为准。

## 缴费水平转换

用户可能输入不同类型的数据，需要统一转换为缴费基数和缴费指数。

| 用户输入 | 转换方式 |
| --- | --- |
| 不清楚，按最低档 | 使用当地缴费基数下限 |
| 税前月薪 | 按当地上下限裁剪为缴费基数 |
| 社保缴费基数 | 直接使用，仍需按上下限校验 |
| 社平工资档位 | 用 60% / 100% / 150% / 300% 转换为缴费指数 |

注意：

- 个人缴费比例通常为 8%，进入个人账户。
- 单位缴费比例通常为 16%，进入统筹账户，不直接进入个人账户。
- 用户最初提到“个人+公司缴纳”，产品可以展示总缴费成本，但养老金公式中要区分个人账户与统筹账户。

## 历史年度参数原则

养老金估算不能把用户过去所有缴费都按最新年份基数计算。

正确原则：

- 用户输入可以是粗略的，例如“长期按 100% 档缴费”。
- 产品自己的城市基础数据必须尽量按年度维护。
- 对历史缴费年份，应使用对应年份、对应城市的社平工资、缴费基数上下限、缴费比例和个人账户记账利率。
- 对退休当年或待遇核定年份，应使用对应年度的养老金计发基数。
- 如果某些历史年份数据缺失，必须标记缺失并采用明确的降级策略，不能静默使用最新年份参数替代。

示例：

```text
用户填写：2006-2025 年长期按 100% 档缴费
系统处理：用 2006、2007、...、2025 每一年上海/北京/深圳对应的年度参数计算，而不是统一用 2025 年参数倒推。
```

这也是“少填也能算，细填更准确”的前提：用户可以少填，但产品底层参数不能偷懒。

## 估算范围

V1 结果建议展示区间，而不是只展示单点值。

示例：

```text
预计每月养老金：约 5200 元
估算范围：约 4400 - 6000 元
```

估算范围可按数据完整度调整，但不应暗示这是统计学置信区间。

| 数据状态 | 展示策略 |
| --- | --- |
| 只填快速测算字段 | 范围较宽，标记为快速估算 |
| 填了个人账户余额 | 范围收窄，标记为账户余额已补充 |
| 历史参数缺失较多 | 范围较宽，并提示部分历史参数仍在补全 |
| 城市参数已复核 | 可展示更高数据质量等级 |
| 涉及多地、视同缴费、特殊退休 | 单独提示精度降低 |

## 竞争力分数目标

竞争力分数用于增强结果页粘性和分享传播，但不是基础测算 V1 的硬依赖。

用户想要类似“打败多少用户”的效果，但 V1 不做登录和真实用户库，所以不能使用真实用户排名。

上线闸门：

- `benchmark-samples.json` 已生成。
- `benchmark-meta.json` 有明确 `benchmarkVersion`。
- 三城基准样本使用的城市参数已达到对应版本的数据门槛。
- 至少 10-20 个样例测试通过，且分数不存在明显反直觉问题。

如果以上条件未满足，结果页和分享页不展示竞争力分数，只展示基础测算结果。

通过闸门后的建议口径：

```text
退休金竞争力 78 分
预计超过 78% 的上海同城估算样本
```

避免默认使用：

```text
打败全国 78% 用户
```

原因：

- V1 没有真实用户数据。
- 全国口径会受到城市、缴费基数、收入水平差异影响，容易误导。
- 分享出去后必须保证稳定、可解释，不能随机变化。

## 固定基准样本

竞争力分数启用后，使用固定基准样本计算百分位。

样本按城市生成：

- 上海样本
- 北京样本
- 深圳样本

样本维度：

- 缴费年限：15 年到 45 年。
- 平均缴费指数：0.6、0.8、1.0、1.2、1.5、2.0、3.0。
- 性别 / 退休年龄类型。
- 城市年度参数。
- 个人账户估算值。

每个样本都使用同一套养老金公式算出 estimatedMonthlyPension。

## 百分位计算

金额百分位：

```text
amount_percentile = 样本中 estimatedMonthlyPension <= 用户 estimatedMonthlyPension 的数量 / 样本总数
```

缴费贡献水平：

```text
contribution_score = 累计缴费年限 × 平均缴费指数
```

缴费贡献百分位：

```text
contribution_percentile = 样本中 contribution_score <= 用户 contribution_score 的数量 / 样本总数
```

缴费年限百分位：

```text
years_percentile = 样本中 contributionYears <= 用户 contributionYears 的数量 / 样本总数
```

个人账户百分位：

```text
account_percentile = 样本中 accountBalance <= 用户 accountBalance 的数量 / 样本总数
```

## 分数公式

如果用户没有填写个人账户余额：

```text
score = 55% × amount_percentile
      + 25% × contribution_percentile
      + 20% × years_percentile
```

如果用户填写了个人账户余额：

```text
score = 50% × amount_percentile
      + 20% × contribution_percentile
      + 15% × years_percentile
      + 15% × account_percentile
```

保护规则：

```text
final_score <= max(contribution_percentile, years_percentile) + 15
final_score 范围限制在 1 到 99
```

保护规则的目的：

- 防止缴费年限短、缴费水平低的人因为某个估算项异常而分数过高。
- 保证分享出去后，缴费更多、年限更长的人大体不会被明显低估。
- 保持分数可解释。

## 竞争力分数展示规则

该部分只在 benchmark 样本通过校验后启用。

数据较少时：

```text
退休金竞争力：约 78 分
```

数据较完整时：

```text
退休金竞争力：78 分
```

推荐说明：

```text
该分数基于你填写的缴费年限、缴费水平、预计养老金和同城估算样本计算。
```

## 分享口径

基础 V1 默认分享：

```text
我用退休金算盘估了一下退休金
少填也能算，细填更准确
```

竞争力分数启用后的分享：

```text
我的退休金竞争力：78 分
预计超过 78% 的上海同城估算样本
```

不默认分享：

- 每月养老金金额。
- 出生年月。
- 个人账户余额。
- 具体缴费记录。

## 模型版本

为了保证分享和结果可追溯，需要给计算模型和样本库设置版本。

示例：

```json
{
  "modelVersion": "pension-abacus-v1",
  "benchmarkVersion": null,
  "cityParamVersion": "2026.05.28"
}
```

当竞争力分数启用后，`benchmarkVersion` 必须更新为明确版本号，例如 `sh-bj-sz-2026-v1`。
