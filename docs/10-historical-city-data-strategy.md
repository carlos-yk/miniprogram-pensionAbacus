# 10 历史城市基础数据策略

更新时间：2026-05-28

本文档补充一个重要原则：用户输入可以粗略，但产品维护的城市基础数据必须尽量完整。

如果我们自己的通用基础数据不完整或口径错误，即使用户认真填写，也无法算准。因此退休金算盘不能只维护最新年份参数，也不能把历史缴费全部按最新年份基数计算。

## 一、核心结论

应该维护历史年度城市基础数据。

原因：

- 基础养老金里的“平均缴费指数”依赖每年缴费基数与当年当地工资口径的比例。
- 个人账户累计额依赖历年个人缴费、记账利率和缴费比例。
- 缴费基数上下限每年变化，不能用最新上下限回推历史。
- 养老金计发基数用于退休待遇核定，一般取退休当年或待遇核定年度参数。
- 历史政策存在分段，例如个人账户比例、过渡性养老金、延迟退休、最低缴费年限等。

产品原则：

```text
用户少填，是交互简化。
基础数据完整，是计算可信度。
两者不能混为一谈。
```

## 二、不能这么算

错误做法：

```text
用户填写：已缴费 25 年，当前 100% 档
系统处理：全部 25 年都按 2025 年社平工资、2025 年缴费上下限、2025 年计发基数计算
```

问题：

- 会夸大历史缴费金额。
- 会扭曲平均缴费指数。
- 会错误估算个人账户累计额。
- 会让不同年龄用户之间的结果失真。

## 三、应该这么算

用户输入可以仍然很简单：

```text
已缴费 25 年
长期按 100% 档缴费
退休城市：上海
```

系统底层应该展开为：

```text
2001 年：使用上海 2001 年工资口径、缴费基数上下限、个人账户比例
2002 年：使用上海 2002 年工资口径、缴费基数上下限、个人账户比例
...
2025 年：使用上海 2025 年工资口径、缴费基数上下限、个人账户比例
```

也就是说，用户粗略选择的是“缴费指数路径”，产品基础库提供每个年份的城市参数。

## 四、历史数据分层

历史数据可以分为三层。

### L1：必须维护

用于 V1 正常测算。

字段：

- 城市
- 年度
- 参数生效期
- 社平工资或全口径平均工资
- 缴费基数下限
- 缴费基数上限
- 职工个人缴费比例
- 个人账户记入比例
- 单位缴费比例
- 数据来源 URL

### L2：强烈建议维护

用于提高准确度。

字段：

- 养老金计发基数
- 个人账户记账利率
- 灵活就业缴费比例
- 灵活就业缴费基数上下限
- 地方过渡性养老金政策分段
- 当年口径说明

### L3：后续精算维护

用于长期精算和多地缴费。

字段：

- 历史政策变更节点
- 本人指数化月平均缴费工资地方算法差异
- 统账结合前后过渡规则
- 地方补贴、补缴政策、断缴处理规则
- 多地转移接续和待遇领取地规则

## 五、V1 支持年份范围建议

上海、北京、深圳 V1 建议先定义支持年份范围。

建议目标：

- 优先回补 2000 年至当前年份。
- 如果可以，继续回补到当地企业职工养老保险制度关键起点。
- 对 1990 年代早期数据，优先用于“视同缴费/过渡性养老金提示”，不强行精算。

原因：

- 大多数 40 岁以上用户的实际缴费集中在 2000 年以后。
- 1990 年代数据口径复杂，地方政策差异大。
- 先保证 2000 年以后质量，比追求一次性全历史更可靠。

## 六、数据缺失降级策略

历史数据缺失时，不能静默使用最新年份参数。

允许的降级策略：

| 缺失情况 | 处理方式 | 结果页提示 |
| --- | --- | --- |
| 某年份社平工资缺失 | 使用相邻官方年份插值或标记缺失 | 当前结果含历史参数估算 |
| 某年份缴费上下限缺失 | 用当年社平工资 60%/300% 估算，并标记 | 部分年份缴费上下限为估算 |
| 个人账户记账利率缺失 | 使用保守默认值或用户填写账户余额覆盖 | 个人账户为估算值 |
| 养老金计发基数缺失 | 不支持该退休年份精算或使用明确回退 | 该城市该年份参数待完善 |
| 过渡性养老金规则缺失 | 不纳入或仅提示 | 过渡性养老金暂未纳入 |

降级结果必须写入计算明细：

```json
{
  "dataQuality": {
    "level": "partial",
    "missingYears": [2001, 2002],
    "estimatedFields": ["contributionBase.floor", "accountInterestRate"],
    "message": "部分历史参数为估算，结果仅供参考。"
  }
}
```

## 七、用户粗输入如何映射到历史参数

用户不需要逐年填写，但系统可以用历史参数展开。

### 场景 1：用户选择长期按 100% 档缴费

系统处理：

```text
每年缴费基数 = 当年当地工资口径 × 100%
再按当年缴费基数上下限裁剪
```

### 场景 2：用户选择长期最低档缴费

系统处理：

```text
每年缴费基数 = 当年当地缴费基数下限
```

### 场景 3：用户填写当前月薪

系统处理：

```text
快速版：推算当前缴费指数，再假设历史平均指数相近
提高准确度：让用户选择“过去比现在低/差不多/更高”
```

### 场景 4：用户填写分阶段缴费

示例：

```text
2005-2015：最低档
2016-2020：100% 档
2021-2025：150% 档
```

系统处理：

```text
按每个阶段逐年套用城市历史参数
```

## 十一、回补优先级

建议按城市和年份分批回补。

第一批：

- 上海 2015-2025
- 北京 2015-2025
- 深圳 2015-2025

第二批：

- 上海 2005-2014
- 北京 2005-2014
- 深圳 2005-2014

第三批：

- 上海 2000-2004
- 北京 2000-2004
- 深圳 2000-2004

第四批：

- 1990 年代关键政策年份
- 主要用于过渡性养老金和视同缴费提示

## 十二、采集方式建议

### V1：人工采集 + 人工复核

适合启动阶段。

步骤：

1. 从官方公告、年鉴、人社局页面收集。
2. 填入 JSON。
3. 复核人逐项检查。
4. 标记数据质量。

### V1.1：半自动检索

用脚本每月搜索官方站点关键词，发现新公告后提醒人工。

不建议自动入库。

### V2：结构化后台

做一个参数管理后台：

- 年份维度。
- 城市维度。
- 字段维度。
- 来源链接。
- 复核状态。
- 发布版本。
- 回滚能力。

## 十三、结果页如何使用数据质量

如果历史数据完整：

```text
当前结果基于上海 2000-2025 年官方历史参数估算。
```

如果部分缺失：

```text
当前结果中部分历史年份参数为估算，建议作为区间参考。
```

如果缺失较多：

```text
该城市历史参数仍在补全中，当前为快速估算，误差可能较大。
```

## 十四、产品结论

需要坚持两件事：

- 用户侧低门槛。
- 数据侧高标准。

这两件事并不矛盾。

用户可以只告诉我们“我大概缴了 25 年，长期 100% 档”，但系统应该用上海、北京、深圳对应年份的历史参数展开计算。这样才符合退休金算盘的产品定位，也能避免工具变成“看起来能算，实际误差很大”的玩具。
